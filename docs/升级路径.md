## 2. 升级路线

### V1.1 - 度量真实化（token 接入）

**目标**：让 `token_cost_total` 指标真实反映 LLM 消耗。

**现状**：`callLlm` 用 `.call().entity(Class)` 直接返回结构化对象，丢弃了 `ChatResponse` 里的 usage 元数据，`recordTokens` 占位 0。

**改法**：
```java
// 当前（占位）
return chatClient.prompt().user(promptText).tools(analysisTools)
    .call().entity(RootCauseAnalysis.class);

// 改造后
ChatResponse response = chatClient.prompt().user(promptText).tools(analysisTools)
    .call().chatResponse();
int tokens = response.metadata().usage().totalTokens().intValue();
metrics.recordTokens(tokens);
// 手动解析 content 为 RootCauseAnalysis（用 ObjectMapper 或 BeanOutputConverter）
return parseRootCause(response.getResult().getOutput().getText());
```

**价值**：token 成本可观测，简历"成本降两个数量级"有真实数据支撑。

---

### V1.2 - L2 真实启用（PgVector）

**目标**：让 L2 向量归并真正工作（替代内存余弦比对）。

**启用步骤**：
1. `application.yml`：`stackwatch.l2.enabled=true`
2. 从 `spring.autoconfigure.exclude` 移除 `DataSourceAutoConfiguration` + `PgVectorStoreAutoConfiguration`
3. 配置 `spring.datasource`（PostgreSQL + `CREATE EXTENSION vector`）
4. 取消注释 `spring.ai.openai.embedding`（EmbeddingModel bean 创建）
5. 建表（见详细设计文档第 6 节）

**待 Spring AI 支持 float[] query 后**：`PgVectorClusterRepository.findSimilar` 从内存余弦切换为 `SearchRequest.builder().query(embedding).topK(1).similarityThreshold(threshold)`。

**价值**：L2 真实向量检索，簇规模可到万级以上。

---

### V1.3 - 采集层 Kafka 真实启用

**目标**：Kafka 削峰 + 解耦 + 重放。

**启用步骤**：
1. `application.yml`：`stackwatch.collector.kafka.enabled=true`
2. 从 `spring.autoconfigure.exclude` 移除 `KafkaAutoConfiguration`
3. 配 `spring.kafka.bootstrap-servers`
4. 启动 Kafka broker + 创建 topic `error-events`

**价值**：异常风暴时削峰，保护分析层；消费失败可重放。

---

### V1.4 - 集成测试（真实 LLM）

**目标**：端到端验证 L3 链路真实跑通。

**现状**：`ErrorAnalyzerTest` 已有 `@EnabledIfEnvironmentVariable(DASHSCOPE_API_KEY)`，配 Key 即可跑。

**补强**：
- 配 API Key 跑 `ErrorAnalyzerTest`，验证真实 LLM 根因质量
- 补"指纹归并质量评估"测试（见 V2.1）

**价值**：眼见为实，面试演示可用。

---

### V2.1 - 指纹算法 V2 + 质量评估

**目标**：指纹算法升级 + 量化归并质量。

**V2 算法**：归一化 volatile 路径与消息 token（如 `OrderService.java:42` -> `OrderService`，消息里的 ID 脱敏），提升相似堆栈聚合率。

**质量评估**（借鉴 PostHog V2）：
- 构建 LLM 标注的异常对数据集（相似/不相似 ground truth）
- 算指纹归并的 precision/recall/F1（pairwise F1）
- V2 上线前确保 F1 > V1（PostHog V2: F1 0.40 vs V1 0.26）

**版本化保证**：`FingerprintVersion.V2` 启用，老簇保持 V1，新簇用 V2，归并关系不断。

**价值**：归并质量可量化，扛住"指纹准不准"的追问。

---

### V2.2 - 反馈飞轮真实存储

**目标**：few-shot 真实持久化 + 向量检索。

**现状**：`InMemoryFewShotRepository` 内存实现，重启丢失。

**升级**：
- `PgVectorFewShotRepository`：存 `few_shot_sample` 表（含 embedding）
- `findByExceptionType` 升级为向量相似检索（找最相似的 few-shot，而非仅按异常类型）
- `FeedbackController` 接入飞书卡片按钮回调（真实反馈入口）

**价值**：数据飞轮真实转起来，越用越准。

---

### V3.1 - 多模型级联（成本优化）

**目标**：粗分用便宜模型，精分用强模型。

**方案**：
- L3 新簇分析：先用 DeepSeek（极低成本）粗分 + 置信度评估
- 置信度 < 0.6 的，再用 Qwen-Plus（强模型）精分
- 缓存：相似堆栈命中缓存不调模型

**价值**：token 成本再降一层，简历"成本优化"更深。

---

### V3.2 - 生产化（L1 Redis + Resilience4j 降级 + 部署）

**目标**：生产级可用性。

**升级项**：
- L1 缓存：Caffeine -> Redis（跨实例共享 + 持久化）
- 降级：Resilience4j 包裹 LLM 调用（超时/熔断/限流），LLM 挂走规则兜底
- 异步化：`ErrorEventCollector.collect` 改投递有界队列异步消费，避免业务线程被 LLM 延迟拖住
- 部署：Docker + K8s，配置中心（Nacos/Apollo）管理 stackwatch.*

**价值**：生产可用，扛住异常风暴 + LLM 故障。

---

### V3.3 - 告警降噪 + 周报增强

**目标**：推送不变成噪音。

**升级项**：
- 告警降噪：稳定低频问题折叠，新簇 + 激增才推送
- 周报 LLM 总结：结构化聚合数据喂 LLM 生成"本周 Top3 + 趋势 + 建议"
- RCA 后动作（借鉴 Aurora）：自动生成 postmortem 草稿 / 建工单

**价值**：从"通知"到"闭环动作"，推送有价值。

---

## 3. 演进原则

1. **渐进式**：每步独立交付，不破坏已跑通链路（L3 单链路 -> L2 -> Kafka -> ...）
2. **条件启用**：新能力用 `@ConditionalOnProperty` 控制，默认关闭，启用才生效
3. **可观测先行**：每个新能力配埋点（Micrometer），效果可量化
4. **不夸大**：未真实接入的（如 token 占位 0）诚实标注 TODO，不谎报数据
5. **借鉴工业界**：指纹版本化（PostHog）、agent 闭环（Aurora）、评估方法论（PyRCA）
