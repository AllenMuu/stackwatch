# StackWatch 详细设计文档

> AI 驱动的 Java 生产错误根因分析系统
> 版本：0.1.0-SNAPSHOT | 最后更新：2026-07-10 | 对应 commit：01aff10

## 1. 项目概述

### 1.1 解决什么问题

生产环境异常堆栈人工定位耗时高、高频问题发现滞后。传统方案依赖研发逐条看日志、复现、定位，存在三个痛点：

- **慢**：从报警到找到根因，人工耗时分钟到小时级
- **散**：同一问题的异常分散在多条日志，难以聚合看趋势
- **无沉淀**：定位经验留在个人脑子里，下次同类问题重复排查

### 1.2 解决方案

将异常堆栈自动归并分类，LLM 进行根因定位，结合定时任务按周聚合高频错误并推送飞书周报。核心是**三层级联归并**：指纹缓存 -> 向量归并 -> LLM，只有约 1% 的异常真正调 LLM。

### 1.3 设计目标

- **低成本**：1% 异常调 LLM，单条分析成本降两个数量级
- **可演进**：指纹版本化、模块条件启用、依赖按需解锁
- **可观测**：监控系统自身有指标（Micrometer + Prometheus）
- **越用越准**：研发反馈回流 few-shot，数据飞轮

## 2. 架构总览

### 2.1 五层主链路 + 两横切

```
① 采集层 Collector       Logback Appender + HTTP /collect + Kafka（默认关闭）
② 预处理层 Preprocessor  指纹去重（SHA-256+框架帧过滤）+ embedding
③ 分析层 Analyzer        L1 指纹缓存 / L2 向量 / L3 LLM 三层级联（核心）
④ 聚合层 Aggregator      实时激增检测 + 按周聚合 Top N
⑤ 投递层 Notifier        飞书 webhook 实时告警 + 飞书周报定时推送

横切 A: 度量层 Metrics   Micrometer：path/耗时/置信度/token，/actuator/prometheus
横切 B: 反馈层 Feedback  研发反馈回流 few-shot，越用越准（数据飞轮）
```

### 2.2 包结构

```
com.stackwatch
├── StackWatchApplication.java        启动类
├── domain/                           不可变 record 数据结构
├── preprocess/                       ②预处理层（Fingerprinter + EmbeddingService）
├── analyzer/                         ③分析层（ErrorAnalyzer + L1/L2/L3 + Tools）
├── collector/                        ①采集层（Logback + HTTP + Kafka）
├── aggregator/                       ④聚合层（周聚合 + 激增检测）
├── notifier/                         ⑤投递层（飞书 + 周报调度）
├── feedback/                         横切B反馈层（few-shot 回流）
├── metrics/                          横切A度量层（Micrometer）
├── config/                           配置（Properties + Bean 装配）
└── web/                              HTTP 入口（Controller + DTO）
```

## 3. 核心概念（domain 层）

所有数据结构用 Java `record`，不可变，compact constructor 做 defensive copy。

| 概念 | 类 | 职责 |
|------|-----|------|
| 错误事件 | `ErrorEvent` | 一次异常的完整信息（类型/消息/堆栈/MDC） |
| 栈帧 | `StackFrame` | 规范化栈帧（类名#方法名，去行号） |
| 指纹 | `ErrorFingerprint` | L1 归并 key（hash + 版本 + topFrames + 可解释记录） |
| 指纹组成 | `FingerprintRecordPart` | 记录指纹由哪几帧/哪部分组成（可调试） |
| 指纹版本 | `FingerprintVersion` | V1/V2，算法升级不断历史归并 |
| 根因分析 | `RootCauseAnalysis` | LLM 结构化输出（根因/分类/置信度/证据/转人工） |
| 错误簇 | `ErrorCluster` | 一组相似异常 + 一次 LLM 归因 |
| 簇状态 | `ClusterStatus` | NEW / ANALYZED / HUMAN_REVIEW |
| 分析路径 | `AnalysisPath` | CACHE_HIT / VECTOR_MERGED / LLM_NEW（埋点用） |
| 分析结果 | `AnalysisResult` | ErrorAnalyzer 返回值（含路径 + 归因） |
| few-shot 样本 | `FewShotSample` | 研发确认的正确根因（反馈飞轮） |

## 4. 核心流程

### 4.1 三层级联归并（③分析层核心）

```
ErrorEvent 进来
  │
  ▼
Fingerprinter.generate(event)  ->  ErrorFingerprint
  │
  ▼
L1: FingerprintCache.lookup(hash)
  ├─ 命中 -> 返回缓存归因（0 token，~1ms）  [AnalysisPath.CACHE_HIT]
  └─ 未命中 ↓
  │
  ▼
EmbeddingService.embed(event)  ->  float[]（L2 关闭时返回 null）
  │
  ▼
L2: ClusterRepository.findSimilar(vec, 0.92)
  ├─ 命中 -> 归入已有簇，计数+1（0 token，~10ms）  [AnalysisPath.VECTOR_MERGED]
  └─ 未命中 ↓
  │
  ▼
L3: LLM 簇代表根因分析（一次调用）
  · buildContext：堆栈 + MDC + few-shot 历史样本
  · Function Calling 工具就绪（查变更/历史/调用链）
  · 结构化输出 RootCauseAnalysis（JSON Schema）
  · postProcess 置信度双判兜底
  · 持久化 + 缓存 + 埋点  [AnalysisPath.LLM_NEW]
```

**成本阶梯**：L1 ≈ 0 token -> L2 ≈ 0 token -> L3 才花钱。真正调 LLM 的只占异常总量约 1%。

### 4.2 错误完整生命周期

```
T0    应用抛出异常
T1    Logback Appender / HTTP /collect / Kafka 捕获  ->  ErrorEvent
T2    Fingerprinter 生成指纹
T3    L1 缓存未命中 -> embedding
T4    L2 向量归并：命中已有簇 / 新簇走 L3
T5    ErrorAnalyzer 返回 AnalysisResult
T6    激增检测：簇计数环比突变 -> 飞书 webhook 即时告警（小时级发现）
T+7d  周报定时任务：按周聚合 Top N + LLM 总结 -> 飞书富文本卡片
T+7d  研发点【根因对/错】-> FewShotSample 回流
T+8d  下次同类堆栈 -> callLlm 注入 few-shot -> 准确率提升
```

### 4.3 反馈飞轮（横切 B）

```
飞书周报卡片（含【根因对】【根因错】按钮）
  └─ 研发点【错】-> 填正确根因
       └─ FeedbackController POST /feedback
            └─ FewShotRepository.save(FewShotSample)
                 └─ 下次 callLlm：fewShotRepository.findByExceptionType()
                      └─ 注入 prompt 的 historicalSamples
                           └─ LLM 基于正确根因样本分析 -> 准确率提升
```

## 5. 关键类职责

### 5.1 分析层（核心）

| 类 | 职责 | 架构层 |
|----|------|--------|
| `ErrorAnalyzer` | 三层级联归并主入口 + 置信度兜底 + 埋点 + few-shot 注入 | ③ L1/L2/L3 |
| `Fingerprinter` | 指纹生成（SHA-256 + 框架帧过滤 + 版本化） | ② 预处理 |
| `EmbeddingService` | embedding 生成（L2 关闭返回 null） | ② 预处理 -> ③ L2 |
| `FingerprintCache` | L1 缓存（Caffeine） | ③ L1 |
| `ClusterRepository` | L2 仓储接口 | ③ L2 |
| `InMemoryClusterRepository` | L2 内存实现（L2 关闭时生效） | ③ L2 |
| `PgVectorClusterRepository` | L2 PgVector 实现（L2 启用时生效，内存余弦比对） | ③ L2 |
| `AnalysisTools` | Function Calling 工具（3 个 @Tool） | ③ L3 |
| `PromptTemplateHolder` | root-cause.st 模板加载与渲染 | ③ L3 |

### 5.2 其他层

| 类 | 职责 |
|----|------|
| `ErrorEventCollector` | 采集服务（Throwable -> ErrorEvent -> analyze） |
| `LogbackErrorAppender` | Logback 自定义 Appender |
| `KafkaErrorProducer/Consumer` | Kafka 采集通道（默认关闭） |
| `WeeklyAggregator` | 按周聚合 Top N |

| `WeeklyReportScheduler` | @Scheduled 周报 + LLM 总结 |

## 6. 数据库表结构（L2 启用时）

当前 L2 默认关闭，用 `InMemoryClusterRepository`。启用 PgVector 后使用以下表结构：

```sql
-- 错误簇表
CREATE TABLE error_cluster (
    cluster_id          VARCHAR(64) PRIMARY KEY,
    representative_hash VARCHAR(64) UNIQUE NOT NULL,
    app_name            VARCHAR(128) NOT NULL,
    exception_type      VARCHAR(256) NOT NULL,
    member_count        INT DEFAULT 1,
    first_seen          TIMESTAMP NOT NULL,
    last_seen           TIMESTAMP NOT NULL,
    status              VARCHAR(32) DEFAULT 'NEW',
    root_cause          TEXT,
    category            VARCHAR(64),
    severity            VARCHAR(16),
    confidence          DOUBLE PRECISION,
    suggested_fix       TEXT,
    need_human_review   BOOLEAN DEFAULT FALSE,
    embedding           VECTOR(1024)
);

-- 簇成员表（指纹 -> 簇）
CREATE TABLE cluster_member (
    fingerprint_hash VARCHAR(64) PRIMARY KEY,
    cluster_id       VARCHAR(64) REFERENCES error_cluster(cluster_id),
    first_seen       TIMESTAMP NOT NULL,
    last_seen        TIMESTAMP NOT NULL,
    occurrence_count INT DEFAULT 1
);

-- few-shot 反馈库
CREATE TABLE few_shot_sample (
    id                 BIGSERIAL PRIMARY KEY,
    cluster_id         VARCHAR(64),
    exception_type     VARCHAR(256),
    stack_text         TEXT NOT NULL,
    correct_root_cause TEXT NOT NULL,
    created_at         TIMESTAMP NOT NULL,
    embedding          VECTOR(1024)
);

-- 向量索引（IVFFlat，cosine）
CREATE INDEX idx_cluster_embedding ON error_cluster
    USING ivfflat (embedding vector_cosine_ops) WITH (lists = 100);
CREATE INDEX idx_fewshot_embedding ON few_shot_sample
    USING ivfflat (embedding vector_cosine_ops) WITH (lists = 100);
```

## 8. 配置（application.yml 关键项）

```yaml
spring:
  autoconfigure:
    exclude:  # 渐进式解锁：未启用模块的自动装配
      - DataSourceAutoConfiguration
      - PgVectorStoreAutoConfiguration
      - KafkaAutoConfiguration
  ai:
    openai:
      api-key: ${DASHSCOPE_API_KEY:}
      base-url: ${LLM_BASE_URL:https://dashscope.aliyuncs.com/compatible-mode/v1}

stackwatch:
  analysis:
    similarity-threshold: 0.92   # L2 相似度阈值
    confidence-threshold: 0.6    # L3 置信度兜底阈值
    fingerprint-top-n: 5
  cache:
    fingerprint-ttl-seconds: 86400
  l2:
    enabled: false               # L2 默认关闭
  collector:
    kafka:
      enabled: false             # Kafka 默认关闭
      topic: error-events
  feishu:
    webhook-url: ${FEISHU_WEBHOOK_URL:}

management:
  endpoints.web.exposure.include: health,info,prometheus
  metrics.tags.application: stackwatch
```

## 9. 模块启用矩阵

| 模块 | 默认 | 启用条件 | 影响 |
|------|------|---------|------|
| L1 缓存 | ✅ 启用 | 始终 | Caffeine 内存 |
| L3 LLM | ✅ 启用 | 配 DASHSCOPE_API_KEY | 真实调 LLM |
| L2 向量归并 | ❌ 关闭 | `l2.enabled=true` + PgVector + datasource + embedding | PgVector 真实检索 |
| Kafka 采集 | ❌ 关闭 | `collector.kafka.enabled=true` + broker + 移除 exclude | Kafka 削峰 |
| 飞书推送 | ⚠️ 占位 | 配 FEISHU_WEBHOOK_URL | 真实推送（留空则 warn 跳过） |

**互斥 bean**：`InMemoryClusterRepository`（`l2.enabled=false`，matchIfMissing=true）vs `PgVectorClusterRepository`（`l2.enabled=true`），同一时刻一个生效。

## 10. 关键设计决策

1. **三层归并而非纯 LLM**：成本降两个数量级，1% 异常才调 LLM（见选型分析文档）
2. **指纹版本化**：算法升级不断历史归并关系（借鉴 PostHog）
3. **Function Calling 防幻觉**：查事实而非猜根因
4. **置信度双判兜底**：LLM 自评 + 证据校验，低置信转人工
5. **渐进式解锁**：`@ConditionalOnProperty` + `autoconfigure.exclude`，零基础设施启动
6. **不可变 record**：所有 domain 用 record，defensive copy
7. **AnalysisPath 埋点**：量化成本故事（95% 缓存 / 4% 归并 / 1% LLM）
