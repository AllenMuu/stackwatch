# StackWatch Agent 工程升级方向

> 借鉴《从 Prompt 到 Harness：企业级 Agent 工程的完整演进之路》一文的工程思想，
> 梳理对 StackWatch **适用 / 不适用** 的升级方向，作为后续升级时对照参考。
>
> 与 [upgrade-path.md](./upgrade-path.md) 的关系：后者是**工程实现路线**（V1.1 token 接入 / V1.2 L2 启用 / V2.1 指纹 V2 …），
> 本文是 **Agent 工程思想视角**的补充——回答"为什么这样升级、哪些思想不该照搬"。
> 两份文档互补：upgrade-path 管"怎么做"，本文管"为什么、值不值得做"。

## 0. 参考来源

- **文章**：《从 Prompt 到 Harness：企业级 Agent 工程的完整演进之路》（阿里千问团队）
- **链接**：<https://mp.weixin.qq.com/s/xH4cyBJJJlG9cfcmSU5ztA>
- **核心主线**：把大模型当成"裸 CPU"，全年工作是为它逐步补一个操作系统——
  `Prompt 工程 -> Context 工程 -> Harness 工程 -> Agent OS`，每一阶段都因前一阶段撞到天花板而出现。
- **一句话定位**：文章讲的是**组织级多 Agent OS**；StackWatch 是**单服务、单 Agent、单步 LLM 调用**系统。
  规模差异决定了"不能照搬，但思想可借鉴"——本文的核心任务就是做这个区分。

## 1. 适用性总判断（先看这张表）

| 文章思想 | 出处 | StackWatch 适用度 | 结论 |
|---|---|---|---|
| 分层防御优于全能方案 | 3.2 / 6.1 | ✅ 已落地，强化叙事 | 代码无需改，升级话术与文档 |
| 确定性优于智能性 / 统计先行 LLM 解读 | 1.3 / 5.4 L3 | ✅ 部分落地，可深化 | 周报/激增检测的统计归因用代码，LLM 只做解读 |
| 可观测性先行 | 4.1 / 6.1 | ✅ 已落地，补维度 | 增加"兜底触发率/飞轮命中率"等驱动指标 |
| 置信度治理 + 分级审批 | 5.4 L3/L4 | ✅ 三档门控已有 | 补"置信度校准"（回溯验证） |
| 反馈飞轮 -> Self-Feedback + Patch 状态机 | 4.4 | ✅ 正负飞轮已有 | 给样本带状态 + 试用期成功率追踪 |
| Action Space 动态裁剪 | 4.1 | ⚠️ 收益有限 | 仅 3 个 @Tool，>10 再考虑 |
| 单一表示原则 / 四层上下文防线 / Prompt 预算预检 | 3.2-3.5 | ⚠️ 暂不需要 | 单步调用，上下文体积小 |
| PERO 多步编排 / 断点续传 / 事件溯源 | 4.2 | ❌ 规模不匹配 | 无长链路，强行套用即过度设计 |
| 多 Agent 树形协调 / Agent OS 五层 / OpenClaw 执行集群 | 5.1-5.2 | ❌ 规模严重不匹配 | 单服务个人项目，无组织级治理需求 |

**判断原则**：StackWatch 的 L3 是**一次 `callLlm` 出 `RootCauseAnalysis`** 的单步推理，不是多步 Agent 循环。
文章里所有为"长链路 / 多步骤 / 多 Agent"设计的机制（上下文压缩、断点续传、多 Agent 协调）都不适用。
真正适用的是**工程审美层**的思想：分层、确定性、可观测、置信度治理、飞轮质量门控。

## 2. 可落地的升级方向

### 2.1 [Context] 把"级联归并"上升为"分层防御"叙事

- **文章出处**：3.2 四层上下文防线、6.1 第二条认知"上下文管理是分层防御的系统工程"
- **StackWatch 现状**：L1 指纹缓存 / L2 向量归并 / L3 LLM 已是三层分级，但文档与面试话术里多停留在"省成本"层面。
- **升级动作**（文档/话术层面，代码无需改）：
  - 把"为什么三层"从"省钱"升级为"不同粒度问题用不同武器"——L1 拦精确重复、L2 拦语义相似、L3 才处理真正的新问题。
  - 沿用文章金句："没有银弹，不同粒度的膨胀需要不同的武器。"
- **价值**：面试时把成本故事讲成工程审美故事，回答"为什么不直接全丢给 LLM"更有说服力。

### 2.2 [Harness] 统计先行，LLM 解读（强化周报与激增检测）

- **文章出处**：5.4 L3 推理层"三步法"——统计归因（代码）-> LLM 叙述解读 -> 置信度标注
- **StackWatch 现状**：
  - 实时激增检测 `HighFrequencyDetector` 是纯计数环比，已是"统计先行"——✅ 符合。
  - 周报 `WeeklyReportScheduler` 直接把 Top-N 聚合数据丢 LLM 总结，**统计归因与 LLM 解读未显式分层**。
- **升级动作**：
  - 周报生成前，先用代码跑结构化统计（环比/同比/新簇占比/分类分布），产出一张"统计事实表"。
  - LLM 只在统计事实表基础上做语义解读（"本周 NPE 激增与 X 模块上线相关"）+ 置信度标注。
  - 降级路径：LLM 失败时直接输出统计事实表（现状已部分如此，强化为"骨架永远在"）。
- **价值**：可信度骨架由统计代码保证，LLM 幻觉不致命——这是"确定性优于智能性"在聚合层的落地。
- **映射**：增强 [upgrade-path.md](./upgrade-path.md) **V3.3 告警降噪 + 周报增强**。

### 2.3 [Harness] 置信度治理：从"三档门控"到"校准 + 分级"

- **文章出处**：5.4 L3 置信度标注、L4 ApprovalGateway（严格 / 阈值 / 事后追认）、5.4 L5 自我评估引擎
- **StackWatch 现状**：`ReviewLevel` 三档门控（`AUTO_CONFIRMED` / `NEEDS_CONFIRMATION` / `NEEDS_HUMAN_REVIEW`）已落地，借鉴 PagePilot。
  这与文章的 ApprovalGateway 分级**思想同构**——可视为 StackWatch 在根因分析垂直领域的收敛（根因分析不存在"拒绝执行"，故三档而非四档）。
- **尚缺的一环：置信度校准**。文章 5.4 L5"自我评估引擎"指出：如果标 `high` 的判断有 40% 是错的，说明模型过度自信，需要校准。
  StackWatch 当前置信度阈值（0.6 / 高置信阈值）是**静态猜测值**，未用真实对错率回溯校准。
- **升级动作**：
  - 利用反馈飞轮的回流数据（研发确认对/错）做回溯验证：统计各置信度区间的实际准确率。
  - 若高置信区间准确率 < 目标（如 90%），自动上调高置信阈值；反之下调。校准周期可按周。
  - 指标化：`置信度校准误差 = 标注置信度 - 实际准确率`，纳入 Micrometer。
- **价值**：阈值不再是拍脑袋，信任建立有数据基础——直接呼应文章 6.1 第五条"信任建立不能跳过人工确认"。
- **映射**：建议作为 [upgrade-path.md](./upgrade-path.md) 新增 **V2.3 置信度校准**（在 V2.2 飞轮真实存储之后，有数据才能校准）。

### 2.4 [Harness] 反馈飞轮升级：Patch 状态机 + 试用期成功率追踪

- **文章出处**：4.4 Self-Feedback Engine、Patch 状态机（`draft -> shadow -> active -> expired`）
- **StackWatch 现状**：`FewShotSample`（正样本）+ `AntiPattern`（负样本）正负双向飞轮已落地。
  但样本目前是**无状态存取**——入库即生效，无试用期、无成功率追踪、无淘汰机制。
- **升级动作**：
  - 给 few-shot / anti-pattern 样本加状态字段：`DRAFT -> SHADOW -> ACTIVE -> EXPIRED`。
    - 新回流样本先进 `SHADOW`：注入 prompt 但不计入"已确认"统计，观察一段窗口的命中表现。
    - 试用期命中率 + 准确率达标的 `SHADOW -> ACTIVE`；持续低成功率（如 100 次命中 <30% 成功）的 `ACTIVE -> EXPIRED`，自动淘汰脏数据。
  - 沿用文章约束"记结论不记过程"：StackWatch 现已只存 `correctRootCause` / `wrongRootCause` 结论文本，不存堆栈 diff——✅ 已符合，可在文档里点明这一契合。
- **价值**：飞轮有质量门控，研发误填的脏样本不会无限污染 prompt；飞轮"越用越准"有机制保障而非口号。
- **映射**：增强 [upgrade-path.md](./upgrade-path.md) **V2.2 反馈飞轮真实存储**——存储升级时一并带上状态机。

### 2.5 [Harness] 可观测性先行：补"驱动优化"维度

- **文章出处**：4.1 渐进式迁移"可观测性先行"、6.1 第四条"工具设计有半衰期"
- **StackWatch 现状**：已有 `path / 耗时 / 置信度 / token` 四类埋点——✅ 基础齐全。
  但缺少"**驱动后续优化优先级**"的维度：不知道哪层兜底触发最频繁、飞轮命中率如何。
- **升级动作**：补三类指标——
  - `review_level_total{level=...}`：三档门控各档计数（量化"转人工率"，驱动 2.3 校准优先级）。
  - `fewshot_hit_total` / `antipattern_hit_total`：飞轮召回命中率（驱动 2.4 状态机升级优先级）。
  - `confidence_fallback_total`：置信度兜底触发次数（量化 LLM 输出质量波动）。
- **价值**：把"可观测性先行"从"有指标"升级为"指标能驱动下一步做什么"——避免盲目优化。
- **映射**：在 [upgrade-path.md](./upgrade-path.md) **V1.1 token 接入**之后，作为 V1.1.5 或并入 V1.1。

## 3. 不适用 / 暂不需要的部分（避免过度设计）

> KISS / YAGNI：以下思想在文章里有价值，但在 StackWatch 当前规模下**强行套用就是过度设计**。
> 每项标注"何时重新评估"，作为未来规模变化时的触发条件。

### 3.1 ❌ PERO 多步编排 / 断点续传 / 事件溯源（4.2）

- **不适用原因**：StackWatch 的 L3 是**单步同步分析**，一次 `callLlm` 出结果，无多步骤状态、无长链路可断点。
  文章的断点续传（15 步任务从重跑 6 分钟到恢复 30 秒）针对的是多步 Agent 循环，StackWatch 没有这个场景。
- **何时重新评估**：若未来引入"多步根因调查 Agent"（如自动复现 -> 变更定位 -> 验证假设的多步推理链）。

### 3.2 ❌ 多 Agent 树形协调 / Agent OS 五层 / OpenClaw 执行集群（5.1-5.2）

- **不适用原因**：单服务个人项目，无组织级治理、多租户、分布式执行需求。
  文章的"CEO -> 主管 -> 执行 Agent"树形协调、L1-L5 操作系统分层是面向组织的，套到 StackWatch 是杀鸡用牛刀。
- **何时重新评估**：项目演化为平台级（多团队接入、多场景 Agent 编排）。

### 3.3 ⚠️ Action Space 动态裁剪（4.1）

- **文章思想**：按步骤进展只暴露当前有意义的工具，减少选择困难。
- **StackWatch 现状**：L3 仅 3 个 @Tool（近期变更 / 历史归因 / 调用链），全量暴露的注意力开销可忽略。
- **结论**：暂不做。工具数 > 10 且观察到 LLM 选错工具率上升时再考虑。

### 3.4 ⚠️ 单一表示原则 / 四层上下文防线 / Prompt 预算预检（3.2-3.5）

- **不适用原因**：单步调用，L3 prompt 体积小（堆栈 + MDC + few-shot + anti-pattern），远未到 128K 窗口压力。
  文章这些机制是为"30+ 步、200K+ 累积膨胀"设计的，StackWatch 没有上下文膨胀问题。
- **何时重新评估**：若 few-shot / anti-pattern 库累积导致 prompt 显著膨胀（如单次注入 > 20 条样本）。
  届时优先借鉴文章 L1"大结果外置 + 引用替换"和 L4"按需预取"。

## 4. 工程审美金句（面试话术储备）

> 来源均为文章原文或其引用，引用时说"业界实践表明"比"事实证明"更稳妥（文章数字是作者经验值，非行业标准）。

1. **"LLM 越跑越蠢是工程问题，不是模型问题。"**
   上下文 70% 噪音稀释注意力，换更大模型只是把天花板从第 5 步推迟到第 8 步。
   *用于 StackWatch*：解释为什么 99% 异常要挡在 LLM 之外，而非全丢给更强模型。

2. **"上下文窗口的物理容量 ≠ 有效容量。"**
   128K 塞满噪音不如精心管理的 32K。
   *用于 StackWatch*：级联归并的根本理由——管信息质量而非扩窗口。

3. **"确定性优于智能性，事前治理优于事后修复。"**
   能用代码的不用 LLM，能用声明式绑定的不用模型搬运。
   *用于 StackWatch*：指纹用 SHA-256、激增检测用计数、根因用 Function Calling 取事实而非猜。

4. **"让 LLM 做它擅长的事（理解意图、规划、推理），让系统做它擅长的事（数据搬运、格式转换、精确传递）。"**
   *用于 StackWatch*：3 个 @Tool 的设计哲学——LLM 只做语义关联，事实由工具取。

5. **"信任建立不能跳过人工确认。"**
   准确率 95% 剩下的 5% 一旦写入数据库，用户立刻失去信任。信任建起来慢（连续 20+ 次正确），毁掉只要一次。
   *用于 StackWatch*：三档门控为何保留 `NEEDS_HUMAN_REVIEW`——不是能力不够，是信任要有梯度。

6. **"Agent 系统设计的本质，是为 LLM 创造一个犯错成本最低、正确路径最短的执行环境。"**
   *用于 StackWatch*：一句话概括整个项目的设计哲学。

## 6. 升级时对照清单

每次升级 L3 / 反馈飞轮 / 周报时，回到本文逐项 check：

- [ ] 升级 L3 置信度相关逻辑时，对照 **2.3**——是否有校准机制？阈值是静态还是数据驱动？
- [ ] 升级反馈飞轮时，对照 **2.4**——样本是否有状态机？低成功率样本是否会自动淘汰？
- [ ] 升级周报/激增检测时，对照 **2.2**——统计归因是否用代码？LLM 是否只做解读？
- [ ] 任何新埋点，对照 **2.5**——这个指标能否驱动下一步优化决策？不能则不埋。
- [ ] 任何想引入文章重型机制（PERO / 多 Agent / 上下文压缩）的冲动，对照 **第 3 节**——规模匹配吗？是否过度设计？
- [ ] 面试/文档表述时，对照 **第 4 节**——用金句把工程审美讲透。
