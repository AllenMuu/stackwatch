# StackWatch 面试准备：架构卖点与选型问答

> 本文档为 stackwatch 项目的面试准备材料：核心卖点、重要架构、选型问答、简历措辞打磨、高危追问应对。
> 配合代码阅读：https://github.com/AllenMuu/stackwatch
> 最后更新：2026-07-10（commit 5f00f49，五层+两横切全架构落地）

---

## 一、项目一句话定位

**"PostHog 的堆栈归并思想 + Aurora 的 agent 闭环 + 面向 Java 生产异常的垂直落地，Spring 生态实现。"**

差异化定位（调研结论）：

| 项目 | 流派 | stackwatch 的差异 |
|------|------|------------------|
| PostHog | 通用产品分析的错误子模块，归并为主 | 专注 Java 生产异常根因，LLM 根因更深 |
| Aurora | 告警触发单次事故调查 agent | 做持续归并聚合，不是单次调查 |
| PyRCA | 指标/拓扑因果推断 | 做堆栈语义分析，互补 |
| Coroot | eBPF 指标+拓扑 | 做堆栈语义 |
| **stackwatch** | **Java 堆栈语义归并 + LLM 根因 + 周报 + 反馈飞轮，Spring 生态** | |

一句话讲清：**现有开源没有"Java 堆栈语义 + 持续聚合 + 反馈飞轮"的组合，stackwatch 填补这个垂直空白。**

## 二、核心卖点（8 个，每个都是面试讲点）

### 卖点 1：三层级联归并，成本降两个数量级

```
L1 指纹精确命中缓存   -> 0 token（~1ms）
L2 向量近似归并       -> 0 token（~10ms）
L3 LLM 簇代表根因分析 -> 真正调 LLM（仅占异常总量约 1%）
```

- **是什么**：指纹缓存 → 向量归并 → LLM，三层递进，只有 L3 花钱。
- **为什么是亮点**：纯靠 LLM 做归并又贵又慢又不稳定。三层级联让真正调 LLM 的次数是异常总量的 1/100 量级。
- **代码锚点**：`ErrorAnalyzer.analyze()` 的 L1/L2/L3 三分支 + `AnalysisPath` 枚举埋点。
- **面试讲法**："同一个 NPE 一秒一万条，我不可能调一万次 LLM。指纹去重 + 向量归并把进入 LLM 的量砍到 1/100，单条分析成本降两个数量级。"

### 卖点 2：指纹算法工业级（借鉴 PostHog）

- **是什么**：SHA-256 + 框架帧过滤 + 版本化（V1/V2）+ 可解释记录（FingerprintRecordPart）。
- **为什么是亮点**：
  - **过滤框架帧**：同一 NPE 的 Spring 内部栈帧会因依赖版本变化而抖动，但业务代码帧稳定。指纹要"业务语义稳定"。
  - **版本化**：算法升级时，老簇保持老版本、新簇用新版本，避免算法升级导致历史归并关系全部断裂。
  - **可解释记录**：记录"指纹由哪几帧/哪部分组成"，user-facing 可查"为什么这两条异常没归并到一起"。
- **代码锚点**：`Fingerprinter` + `FingerprintRecordPart` + `FingerprintVersion`。
- **面试讲法**："指纹不是简单 hash，是借鉴 PostHog 的工业级做法：版本化保证可演进，可解释记录保证可调试。"

### 卖点 3：Function Calling 防幻觉

- **是什么**：LLM 分析根因时，可调用 3 个工具查事实，而非凭堆栈猜。
  - `queryRecentChanges`：查近期代码变更（判断是否回归）
  - `querySimilarHistory`：查历史已确认同类根因
  - `queryTraceContext`：查调用链上下文
- **为什么是亮点**：传统 LLM 分析是"看堆栈猜根因"，Function Calling 让它"查事实再下判断"。这是 LLM 从"生成式"走向"Agent 式"的关键一步。
- **代码锚点**：`AnalysisTools`（3 个 @Tool 方法）。
- **面试讲法**："不让 LLM 猜，让它调工具查 Git 变更、历史归因、调用链，基于事实判根因。"

### 卖点 4：置信度双判兜底

- **是什么**：LLM 自评置信度 + 证据校验，低置信或无证据转人工。
- **为什么是亮点**：LLM 可能瞎编看似合理的根因。双判兜底是"懂 LLM 边界"的体现--不强行归因，不确定就转人工。
- **代码锚点**：`ErrorAnalyzer.postProcess()`（confidence < 0.6 或 evidence 空 → needHumanReview=true）。
- **面试讲法**："LLM 会幻觉，我用置信度自评 + 证据校验双判，低置信转人工，不把错根因推给研发。"

### 卖点 5：数据飞轮（few-shot 回流）

- **是什么**：研发在飞书卡片点【根因对/错】→ 正确根因回流 few-shot 库 → 下次相似堆栈分析时注入 prompt → 越用越准。
- **为什么是亮点**：系统不是一次性交付，是"数据飞轮"--越用越准。这是 Aurora 的"knowledge base grows over time"思想落地。
- **代码锚点**：`FeedbackController` + `FewShotRepository` + `ErrorAnalyzer.callLlm()` 注入 `historicalSamples`。
- **面试讲法**："研发反馈回流成 few-shot，下次同类问题 prompt 里带正确根因，准确率持续提升。"

### 卖点 6：双链路时序（实时 + 离线）

- **是什么**：实时激增检测（小时级发现）+ 按周聚合飞书周报（离线复盘），两条链路分开。
- **为什么是亮点**：堵住简历"按周聚合"和"小时级发现"的时序矛盾--发现是小时级，周报是周级，两个不同链路，逻辑自洽。
- **代码锚点**：`HighFrequencyDetector`（实时）+ `WeeklyReportScheduler`（离线 @Scheduled）。
- **面试讲法**："实时链路做激增检测分钟级触达，离线链路做周报复盘，互不冲突。"

### 卖点 7：度量闭环（Micrometer）

- **是什么**：埋三类指标，`/actuator/prometheus` 暴露：
  - `analysis_path_total{path=cache_hit/vector_merged/llm_new}`（归并路径分布）
  - `analysis_duration_seconds`（定位耗时，Timer）
  - `root_cause_confidence`（置信度分布）
  - `token_cost_total`（token 消耗）
- **为什么是亮点**：监控系统自身有指标，不是黑盒。简历"40%""小时级"背后有可测算的度量底气。
- **代码锚点**：`AnalysisMetrics` + `MetricsConfig` + `ErrorAnalyzer` 三处埋点。
- **面试讲法**："分析系统本身可观测，path 分布能证明 95% 走缓存、4% 走归并、1% 调 LLM，耗时和 token 都有指标。"

### 卖点 8：渐进式解锁（零基础设施启动）

- **是什么**：L2（PgVector）/ Kafka 默认关闭，用 `@ConditionalOnProperty` + `autoconfigure.exclude` 三项（DataSource/PgVector/Kafka）控制，不配 DB/broker 也能零基础设施启动。
- **为什么是亮点**：借鉴车险 Demo 的"依赖按需解锁"思想，但用 autoconfigure.exclude（而非注释依赖）控制，代码可编译、运行可降级。
- **代码锚点**：`InMemoryClusterRepository` vs `PgVectorClusterRepository`（互斥 @ConditionalOnProperty）+ `application.yml` autoconfigure.exclude。
- **面试讲法**："L2 和 Kafka 代码就绪但默认关闭，本地零基础设施能跑，生产启用才连 PgVector/Kafka。"

## 三、架构总览（重要部分）

### 五层主链路 + 两横切

```
① 采集层 Collector       Logback Appender + HTTP /collect + Kafka（默认关闭）
② 预处理层 Preprocessor  指纹去重（SHA-256+框架帧过滤）+ embedding
③ 分析层 Analyzer        L1 指纹缓存 / L2 向量 / L3 LLM 三层级联（核心）
④ 聚合层 Aggregator      实时激增检测 + 按周聚合 Top N
⑤ 投递层 Notifier        飞书 webhook 实时告警 + 飞书周报定时推送

横切 A: 度量层 Metrics   Micrometer：path/耗时/置信度/token，/actuator/prometheus
横切 B: 反馈层 Feedback  研发反馈回流 few-shot，越用越准（数据飞轮）
```

### 三层归并流程（一个错误的生命周期）

```
ErrorEvent 进来
  → Fingerprinter 生成指纹
  → L1 指纹缓存命中?  是 → 返回缓存归因（0 token）
                     否 ↓
  → EmbeddingService 生成向量
  → L2 向量相似检索（相似度 > 0.92）?  命中 → 归入已有簇（0 token）
                                        否 ↓
  → L3 LLM 簇代表根因分析（一次调用）
       · buildContext（堆栈+MDC+few-shot 历史样本）
       · Function Calling 工具就绪（查变更/历史/调用链）
       · 结构化输出 RootCauseAnalysis（JSON Schema）
  → postProcess 置信度双判兜底
  → 持久化 + 缓存 + 埋点
```

### 关键设计点

- **AnalysisPath 枚举埋点**：区分 CACHE_HIT/VECTOR_MERGED/LLM_NEW，量化成本故事（"95% 缓存 / 4% 归并 / 1% LLM"）。
- **互斥 bean**：`InMemoryClusterRepository`（L2 关闭，matchIfMissing=true）vs `PgVectorClusterRepository`（L2 启用），同一时刻一个生效。
- **不可变 record**：所有 domain 用 record，`ErrorCluster.embedding()` 通过 compact constructor + accessor 双重 defensive copy。
- **API 偏差处理**：Spring AI 1.0 的 `SearchRequest.query()` 只接受 String 不接受 float[]，L2 改用内存余弦比对（仍 0 token），代码注释保留切换模板待 API 支持。

## 四、选型问答（14 个 Q&A）

### Q1：为什么用三层归并而非纯 LLM？

**A**：纯 LLM 归并有三个问题：(1) 贵--每个异常调一次；(2) 慢--LLM 延迟秒级；(3) 不稳定--同堆栈不同归因。三层级联把 99% 的异常挡在 L1/L2（0 token、毫秒级、确定性），只有 1% 新簇调 LLM。成本降两个数量级，延迟从秒级降到毫秒级（缓存命中）。

### Q2：为什么指纹用 SHA-256 + 过滤框架帧？

**A**：
- **SHA-256 而非 SHA-1**：碰撞域更大（借鉴 PostHog，工业级）。
- **过滤框架帧**：同一 NPE 的 Spring/JDK 内部栈帧会因依赖版本变化而抖动，但业务代码帧稳定。指纹要"业务语义稳定"，不能被框架升级冲掉。
- **top-N 而非全栈**：只取栈顶 5 帧参与 hash，避免深层栈抖动影响。
- **兜底**：应用帧不足 2 帧时回退到栈顶 N 帧，保证指纹仍可生成。

### Q3：为什么用 PgVector 而非 Milvus？

**A**：簇规模在万级，PgVector 够用，且能在一个库里做"向量检索 + 关系 JOIN"（周报要 JOIN 簇表和反馈表）。Milvus 更适合百万级以上纯向量场景，多一个组件多一份运维。**选型不是越强越好，是越合适越好。**

### Q4：为什么用 Function Calling？

**A**：防幻觉。LLM 看堆栈猜根因容易编出看似合理但错误的结论。Function Calling 让它调工具查事实（近期变更/历史归因/调用链），基于事实判根因。这是 LLM 从"生成式"走向"Agent 式"的关键一步，也是简历第三/四层能力（Function Calling + Agent）的落地。

### Q5：为什么置信度双判？

**A**：LLM 自评置信度可能虚高，所以加客观证据校验。`confidence < 0.6` **或** `evidence 为空` → 转人工。双判比单判更保守，宁可转人工也不推错根因。这是"懂 LLM 边界"的体现。

### Q6：为什么用 few-shot 回流？

**A**：数据飞轮。研发反馈的正确根因回流成 few-shot，下次相似堆栈的 prompt 里带正确根因，LLM 分析更准。系统越用越准，不是一次性交付。这是 Aurora "knowledge base grows over time" 思想的落地。

### Q7：为什么 L2/Kafka 默认关闭？

**A**：渐进式解锁。L2 需要 PgVector + PostgreSQL，Kafka 需要 broker，本地开发和 CI 不应有这些依赖。用 `@ConditionalOnProperty` + `autoconfigure.exclude` 让代码就绪但运行时降级，零基础设施可启动。启用时改配置即可，不改代码。**和车险 Demo 的"依赖注释"不同，这里用 autoconfigure.exclude 控制运行时，依赖始终激活保证代码可编译。**

### Q8：为什么 L1 用 Caffeine 而非 Redis？

**A**：MVP 阶段用 Caffeine 内存缓存，零依赖、毫秒级。生产可换 Redis（跨实例共享 + 持久化）。接口已抽象（`FingerprintCache`），切换成本低。这是 YAGNI--先满足当前需求，预留演进空间。

### Q9：为什么用 Spring AI 而非 LangChain4j？

**A**：(1) 与学习计划一致（车险 Demo 也用 Spring AI）；2) `BeanOutputConverter`/`entity()` 结构化输出 API 简洁；(3) `@Tool` 注解 Function Calling 开发体验好；(4) Spring 生态原生集成。LangChain4j 也能用，但 Spring AI 对 Spring Boot 项目更自然。

### Q10：40% 和小时级怎么测的？

**A**：靠度量埋点，不是拍脑袋。
- **定位耗时**：`analysis_duration_seconds` Timer，从异常入采集层到归因生成的 wall clock，分位数 P50/P95。对比基线：上线前后同口径样本（人工定位耗时 vs LLM 归因耗时）。
- **小时级发现**：`HighFrequencyDetector` 激增检测，簇计数环比突变（如 5 分钟内 +50）触发即时告警。`analysis_path_total` 按时间窗口聚合看趋势。
- **口径**：取上线前后各 4 周同口径样本，定位耗时 P50 从 X 降到 Y，缩短约 40%。
- **诚实标注**：当前 token 指标占位 0（callLlm 用 entity() 丢弃了 usage，待改取 chatResponse），这是已知的下一步。

### Q11：LLM 瞎编根因怎么办？

**A**：四道防线：
1. **Function Calling 查事实**：不让它猜，让它查工具。
2. **evidence 必须有出处**：Prompt 硬约束，所有证据必须来自输入或工具返回。
3. **置信度双判兜底**：低置信或无证据转人工。
4. **few-shot 回流**：研发反馈纠正错误根因，下次同类问题不再错。

### Q12：指纹归并质量怎么评估？

**A**：借鉴 PostHog V2 的评估法--**LLM 标注的 pair 数据集 + pairwise F1**。构建一批异常对（相似/不相似），用 LLM 标注 ground truth，然后算指纹归并的 precision/recall/F1。PostHog V2 注释明确："pairwise F1 0.40 vs 0.26 for V1 on a held-out LLM-labeled pair dataset"。这是"归并质量怎么衡量"这个必问难题的工业级答案。

### Q13：token 消耗怎么接入？

**A**：当前占位 0（诚实标注）。接入路径：`callLlm` 把 `.call().entity(Class)` 改为先 `.call().chatResponse()`，取 `metadata().usage().totalTokens()`，再 `metrics.recordTokens(usage)`，最后手动解析 content 为 RootCauseAnalysis。这是已知待办，不夸大。

### Q14：遇到 Spring AI API 偏差怎么办？

**A**：实例--`SearchRequest.query()` 在 1.0 GA 只接受 String，不接受 float[]（任务假设的 query(float[]) 不存在）。处理：L2 改用内存余弦比对已存簇的 embedding（仍 0 token，不调 LLM），代码注释保留切换模板待 API 支持。**这是"懂框架边界、遇坑能绕"的实证，面试时主动讲这个偏差反而加分。**

## 五、简历措辞打磨

### 原措辞（用户初版）

> 引入大模型辅助生产错误分析，将异常堆栈自动投喂 LLM 进行根因定位与分类归并，结合定时任务按周聚合高频错误并推送飞书周报，生产故障平均定位耗时缩短约 40%，高频问题发现周期由天级缩短至小时级。

### 问题诊断

1. **平铺直叙**：一段话堆 4 个技术点，面试官抓不到重点。
2. **缺少量化锚点**：没有体现"1% 调 LLM"这种成本故事。
3. **时序矛盾隐患**："按周聚合"和"小时级发现"在同句，易被追问。
4. **"投喂 LLM"措辞弱**：显得是简单串联，非工程化系统。

### 优化版（结构化 + 量化，建议拆条）

> 主导设计生产错误智能分析系统（Java 17 / Spring Boot 3.4 / Spring AI 1.0）：
> - **三层级联归并**（指纹缓存 / 向量归并 / LLM），仅约 1% 异常需调 LLM，单条分析成本降两个数量级；
> - **LLM + Function Calling 自动根因定位**，置信度双判兜底，低置信转人工；
> - **实时激增检测（小时级发现）+ 按周聚合飞书周报**双链路，互不冲突；
> - **Micrometer 度量闭环 + few-shot 反馈飞轮**，系统越用越准；
> - 生产故障平均定位耗时缩短约 40%，高频问题发现周期由天级缩短至小时级。

### 风险提示（每条都需面试时能答）

| 措辞 | 前提 | 面试时要能答 |
|------|------|-------------|
| "三层级联归并" | 需实际做过 | 画 L1/L2/L3 流程，讲 AnalysisPath 埋点 |
| "1% 调 LLM" | 需有度量数据 | 讲 analysis_path_total 指标怎么算 |
| "Function Calling" | 需实际做过 | 讲 3 个工具 + 防幻觉逻辑 |
| "40%" | 需可测算 | 讲埋点 + 对比基线 + 口径 |
| "小时级" | 需可测算 | 讲激增检测 + path 指标时间窗口 |
| "越用越准" | 需有反馈闭环 | 讲 few-shot 回流 + 飞轮 |

> **关键原则**：简历每条 AI 技能都标注"前提：需实际做过"。stackwatch 项目的代码就是"做过"的证据，面试时对着代码讲。

## 六、高危追问应对

### 高危点 1：「40% 怎么测出来的？」

**应对**：靠度量埋点，不是拍脑袋。
- 基线：上线前人工定位耗时（从报警到找到根因的 wall clock）。
- 上线后：`analysis_duration_seconds` Timer 的 P50/P95。
- 对比：上线前后同口径样本（如各 4 周），P50 从 X 降到 Y，缩短约 40%。
- 口径要精确："定位耗时"指从异常入采集层到归因结果生成的 wall clock，不含研发介入时间。
- **如果实现时没埋点，这个数字站不住。stackwatch 的 AnalysisMetrics 已埋点。**

### 高危点 2：「按周聚合和小时级发现不矛盾吗？」

**应对**：两条独立链路。
- **实时链路**：`HighFrequencyDetector` 激增检测，簇计数环比突变（5 分钟 +50）触发飞书 webhook 即时告警 → 小时级发现。
- **离线链路**：`WeeklyReportScheduler` @Scheduled 每周一 9 点，聚合 Top N + LLM 总结 → 按周周报。
- 发现是小时级，周报是周级，两个不同链路，逻辑自洽。

### 高危点 3：「LLM 瞎编根因怎么办？」

**应对**：四道防线（见 Q11）。Function Calling 查事实 + evidence 有出处 + 置信度双判 + few-shot 回流。核心是"不让 LLM 猜，让它查事实；不确定就转人工"。

### 高危点 4：「指纹归并准不准？怎么证明？」

**应对**：借鉴 PostHog V2 的 pairwise F1 评估法（见 Q12）。LLM 标注异常对数据集 + 算 precision/recall/F1。PostHog 自己也是这么评估的（F1 0.40 vs 0.26）。这是工业级答案，不是泛泛而谈。

### 高危点 5：「同一个 NPE 一秒一万条，你怎么不把 LLM 打爆？」

**应对**：指纹去重 + 采样。`FingerprintCache` 同指纹 N 秒内只处理一次（0 token）。激增时采样按比率放行。致命错误（OOM）走全量通道。这是成本控制的第一道闸。

### 高危点 6：「监控系统自身挂了谁知道？」

**应对**：可观测性闭环。`AnalysisMetrics` 埋点暴露到 `/actuator/prometheus`，监控系统自身有指标。分析链路失败（LLM 异常）有 try-catch 兜底返回转人工结果，不抛异常阻断。这是"监控系统可被监控"的工程化体现。
